_Research date: 2026-04-05_

This document records best-practice findings for each of the 10 review agent domains. Each finding is ranked highest-impact first and includes an inline source.

### 10k-foot

The 10k-foot agent ensures that an implementation plan aligns with the system's existing architecture, stated goals, and desired end-product before any code is written.

1. Architectural impact must be evaluated independently from code correctness — code can pass all unit tests and style checks yet still cause long-term structural damage if its architectural fit is not assessed. Teams should define complexity thresholds: solo review for utilities, pair review for business logic, and formal architecture review for core integrations. (https://www.codeant.ai/blogs/code-review-process-guide)
2. A living architectural checklist, reviewed quarterly against recent production incidents and new patterns, prevents drift from accumulating silently across multiple sessions — outdated review criteria produce false passes on plans that introduce subtle misalignment. (https://www.kluster.ai/blog/code-review-checklist)
3. High-level threat modeling and abuse-case analysis should be integrated at the plan level, not deferred to implementation, so that security and scalability anti-patterns are caught before structural decisions become load-bearing. (https://cheatsheetseries.owasp.org/cheatsheets/Secure_Code_Review_Cheat_Sheet.html)
4. AI-generated or AI-assisted plans require human reviewers to shift focus to architectural integrity and business intent, since automated tools excel at style and syntax but routinely miss whether a proposed structure fits the system's long-term direction. (https://www.metacto.com/blogs/establishing-code-review-standards-for-ai-generated-code)
5. Downstream consumers — API clients, UI components, and dependent services — must be named explicitly in the plan; omitting them is the most common cause of plans that appear complete but leave the system in a partially integrated state. (https://graphite.com/guides/ai-code-review-implementation-best-practices)

### technical-expert

The technical-expert agent performs deep implementation-level review: surfacing known library pitfalls, version incompatibilities, incorrect ordering of steps, and missing integration requirements before execution begins.

1. Reviewers must read code as a potential attacker and as a future maintainer simultaneously — looking for edge cases, concurrency issues, and design flaws that are invisible when reading only for "does this do what it says." The mix of expertise across reviewers (security, performance, architecture) is what catches issues that a single-perspective review misses. (https://google.github.io/eng-practices/review/reviewer/looking-for.html)
2. A common and costly mistake is over-emphasizing style during review while under-investing in logic correctness and integration fit — reviewers should spend the bulk of their attention on whether the described steps will actually produce a working integration, not on formatting. (https://graphite.com/guides/common-code-review-mistakes-to-avoid)
3. Implementation ordering errors (e.g., schema migration before model update, cache registration before the cache client is initialized) are the most common source of session failures; reviewers must trace the described execution sequence against the technology's required initialization order. (https://axolo.co/blog/p/common-code-review-mistakes-developers-make)
4. Integration gaps between sessions — where two components must communicate but no session defines the interface — are underdetected because each session looks reasonable in isolation; reviewers must explicitly trace data flow across session boundaries to find these gaps. (https://www.wiz.io/academy/application-security/code-review-best-practices)
5. Reviewer attention degrades sharply after 60–90 minutes of focused review; deep technical review of complex plans should be broken into focused segments to maintain the error-detection rate that makes the review valuable. (https://www.aikido.dev/blog/common-code-review-mistakes)

### scope-limiter

The scope-limiter agent detects when implementation sessions contain more work than their stated goal, flags incorrect session ordering, and identifies parallelism opportunities.

1. Scope creep is an anti-pattern where unplanned features are added to work-in-progress without adjusting timelines or resources; during plan review, every session must be checked for files modified outside its stated scope, features not present in the spec, and "drive-by" refactoring. (https://www.minware.com/guide/anti-patterns/scope-creep)
2. Internal "gold-plating" — where team members add unrequested enhancements — is distinct from stakeholder-driven scope creep and is harder to detect because each addition seems individually reasonable; reviewers must compare each session's deliverables against the session's single stated goal. (https://deviq.com/antipatterns/feature-creep/)
3. Concrete red-flag questions that reliably surface scope violations during review: "Why was this unrelated file changed?", "This feature wasn't in the spec — who approved it?", and "Can this enhancement become a separate session?" — making the check explicit and repeatable prevents it from being skipped. (https://docs.gitscrum.com/en/best-practices/preventing-scope-creep-in-agile-projects)
4. Combining two logically independent deliverables in one session raises risk without reducing total work; splitting them into separate sessions reduces blast radius and makes rollback feasible if one deliverable fails. (https://eastwall.com/blog/mitigating-scope-creep-in-software-projects)
5. Code review anti-patterns that AI tools eliminate most effectively include missed scope violations and ordering inconsistencies — automated scope checking is high-value precisely because human reviewers habituate to the plan's structure and stop noticing when a session has quietly expanded. (https://www.coderabbit.ai/blog/5-code-review-anti-patterns-you-can-eliminate-with-ai)

### test-coverage

The test-coverage agent verifies that the plan describes tests of the right type and granularity for every significant behavior, with coverage proportional to risk and no fragility or flakiness baked in by design.

1. The testing pyramid — 70% unit, 20% integration, 10% end-to-end — represents the industry consensus on cost-effective coverage distribution; plans that invert this pyramid (heavy E2E, sparse unit) produce slow, expensive, and brittle test suites that break during normal refactoring. (https://shiftasia.com/column/unit-integration-e2e-testing-guide/)
2. The most common and damaging testing mistake is over-relying on E2E tests for features that integration or unit tests could cover more cheaply — E2E tests should be reserved for critical user workflows where the only meaningful assertion is that the whole system works together. (https://www.bunnyshell.com/blog/best-practices-for-end-to-end-testing-in-2025/)
3. Integration tests catch interface mismatches that unit tests structurally cannot — a plan that tests each component in isolation but omits integration tests for their shared boundaries will pass review and fail in production when the components are connected. (https://www.getautonoma.com/blog/integration-vs-e2e-testing)
4. Test code must be reviewed with the same rigor as production code; plans that describe tests at a level too abstract to be actionable ("verify it works") provide false confidence and will produce tests that pass even when the feature is broken. (https://www.graphite.com/guides/code-coverage-best-practices)
5. ML-assisted coverage analysis can identify untested high-risk code paths based on historical defects and complexity metrics — plans should explicitly note which behaviors are high-risk and confirm that coverage addresses happy path, error path, and edge cases for each. (https://www.mabl.com/blog/defining-good-test-coverage-with-unit-testing-and-end-to-end-testing)

### security

The security agent reviews implementation plans for attack surface expansion, access control gaps, data exposure risks, secrets handling, and input validation before any code is written.

1. The OWASP Secure Code Review Guide prescribes integrating threat modeling and abuse-case analysis at the plan level, not the code level — plans that defer security review to implementation routinely miss architectural decisions that cannot be safely corrected after the fact. (https://owasp.org/www-project-code-review-guide/)
2. Authorization checks described only at the UI layer (hiding buttons for non-admins) with no corresponding server-side enforcement is the single most common authorization gap in implementation plans; every privileged action must specify server-side enforcement explicitly. (https://cheatsheetseries.owasp.org/cheatsheets/Secure_Code_Review_Cheat_Sheet.html)
3. Every new entry point introduced by a plan — endpoint, webhook, file upload, message queue consumer — must specify authentication requirement, authorization check, rate limiting, and input validation; missing any one of these is a reviewable flag regardless of what the other three say. (https://owasp.org/www-project-secure-coding-practices-quick-reference-guide/)
4. Secrets described as stored in code, configuration files, or any location tracked by version control is a blocking finding; the plan must name the secrets management approach (environment variables, secrets manager) for every credential it introduces. (https://www.awesomecodereviews.com/checklists/secure-code-review-checklist/)
5. Using a review checklist that maps directly to the OWASP secure coding checklist makes security review repeatable and auditable — ad-hoc security review produces inconsistent coverage and gives teams false confidence that security was addressed. (https://www.appsecmaster.net/blog/secure-coding-practices-checklist-for-owasp-top-10/)

### data-model

The data-model agent reviews schema changes, migration scripts, and data lifecycle decisions to ensure correctness, migration safety, backwards compatibility, and no silent data loss.

1. The expand-contract pattern — add new schema elements alongside existing ones, migrate data, then remove old elements — is the industry-standard approach for zero-downtime schema changes; plans that drop or rename columns without this pattern are a blocking finding because they will break the running application during deployment. (https://planetscale.com/blog/backward-compatible-databases-changes)
2. Adding a NOT NULL constraint to an existing column that may contain nulls, or narrowing a field's type, are the two most common backwards-incompatible schema changes reviewers miss because they look like minor additions; each requires a backfill migration and must be flagged as critical. (https://www.pingcap.com/article/database-design-patterns-for-ensuring-backward-compatibility/)
3. Migrations must be tested in staging against both old and new application code simultaneously — a migration that is safe under new code can corrupt data when old code is still serving traffic during a rolling deploy, a scenario that plan reviewers must explicitly check for. (https://planetscale.com/blog/safely-making-database-schema-changes)
4. Monetary amounts stored as floating-point types introduce silent rounding errors that compound over time; plans introducing financial or quantity fields must use decimal or numeric types, and this is a correctness issue, not a style issue. (https://logisam.com/7-essential-database-migration-best-practices-for-2025/)
5. Automated schema linting that checks for backwards-incompatible changes should be integrated into the plan review process — visual schema diffs showing exactly which tables and columns are created, altered, or dropped make the migration surface reviewable without relying solely on prose descriptions. (https://blog.jetbrains.com/idea/2025/02/database-migrations-in-the-real-world/)

### api-contract

The api-contract agent ensures that every interface introduced or modified by the plan makes a complete, consistent, and durable promise to its consumers, including error contracts and versioning strategy.

1. Breaking changes — removing a field, changing a field's type, making an optional field required — must trigger a versioning strategy (path versioning, feature flag, deprecation window); plans that modify existing interfaces without addressing consumers are a blocking finding when those consumers are known. (https://zuplo.com/learning-center/api-versioning-backward-compatibility-best-practices)
2. Using a standard error structure such as RFC 7807 ProblemDetails for all error responses prevents the inconsistency — some endpoints returning `{error: "..."}`, others returning HTTP codes with no body — that makes client error handling brittle and untestable. (https://agileseekers.com/blog/handling-backward-compatibility-in-versioned-product-apis)
3. Contract testing — establishing formal consumer-provider agreements and validating them automatically in CI — is the only mechanism that reliably catches interface regressions before they reach production; plans that introduce interfaces without specifying how contracts will be captured and tested leave consumers unprotected. (https://medium.com/qualitynexus/api-versioning-and-backward-compatibility-complete-testing-guide-for-quality-engineers-669d46d204d7)
4. API consumers should be notified of breaking changes 6–12 months in advance with gradual migration paths; plans that describe interface changes without a corresponding consumer update session are incomplete regardless of how well the producer side is specified. (https://www.gravitee.io/blog/api-versioning-best-practices)
5. Naming inconsistencies across interfaces in the same plan — `camelCase` in one endpoint, `snake_case` in another, `user_id` vs `userId` vs `account_id` for the same concept — are the most common internal consistency failure and the hardest to correct after consumers have been built against the inconsistent contracts. (https://www.speakeasy.com/api-design/versioning)

### performance

The performance agent reviews structural decisions in the plan that tend to produce slow, resource-hungry, or fragile systems under real-world load, flagging issues before they are built in.

1. The N+1 query pattern — one query to fetch a list, then one query per item to fetch related data — is the single most common plan-level performance anti-pattern; it appears in plans as innocuous loops over collection results and scales catastrophically (100 records = 101 queries). (https://medium.com/@artemkhrenov/query-optimization-patterns-writing-efficient-sql-for-high-performance-applications-8143e5028443)
2. The no-caching anti-pattern — repeatedly fetching the same data from expensive resources on every request — is the second most common structural performance problem; plans must specify caching scope (per-user, shared), invalidation strategy, and TTL for any data that is read frequently and changes rarely. (https://learn.microsoft.com/en-us/azure/architecture/antipatterns/no-caching/)
3. Slow operations placed synchronously in user-facing request paths — sending email, calling external APIs, processing files, generating reports — should be moved to background queues; plans that describe these as synchronous are a flag because the fix requires an architectural change, not just a code change. (https://www.infoq.com/articles/Diagnose-Microservice-Performance-Anti-Patterns/)
4. Plans without defined performance targets (response time, throughput, max memory for batch jobs) have no way to determine during execution whether performance is acceptable or to catch regressions later; performance targets are a plan-level requirement, not a post-deployment concern. (https://redis.io/tutorials/redis-anti-patterns-every-developer-should-avoid/)
5. Missing database indexes on columns used in search, sort, or join operations cause full-table scans that appear only under production data volumes; plans introducing queries must explicitly name the indexes that support them, not assume the ORM will handle it. (https://dev.to/cristiansifuentes/10-sql-anti-patterns-you-must-avoid-in-production-4m40)

### dependency

The dependency agent evaluates every new external library, service, or infrastructure component the plan introduces for necessity, license compatibility, maintenance health, version pinning, and runtime compatibility.

1. License compatibility is the highest-priority dependency check because it is a blocking legal risk: GPL and AGPL dependencies in a commercial product typically require the entire product to be released under the same license; plans must name the license of every new dependency and confirm it is compatible with the project's commercial use. (https://cloud.google.com/blog/topics/developers-practitioners/best-practices-dependency-management)
2. Exact version pinning — specifying precise version numbers rather than ranges like `^1.2.0` — combined with a committed lock file is the minimum standard for reproducible builds; unpinned dependencies mean a new release can silently introduce breaking changes or vulnerabilities between plan approval and execution. (https://dev.to/mlamina/why-pinning-your-dependency-versions-matters-n24)
3. Dependency health assessment must cover maintenance activity (releases in the past 12–18 months), issue backlog, ownership model (single maintainer vs. organization), and existence of a migration path if abandoned — single-maintainer libraries with no recent activity are a risk the plan must explicitly acknowledge. (https://tss-yonder.com/insights/best-practices-for-dependency-management)
4. Tools like OWASP Dependency-Check and GitHub Dependabot should be part of the review process; plans that introduce dependencies without running them through vulnerability scanning are accepting unknown security risk that automated tools would surface in minutes. (https://www.packetlabs.net/posts/policies-processes-and-tools-for-managing-software-dependency-security/)
5. A dependency introduced for a narrow use case should be challenged against alternatives: something already in the project's dependency tree, a small amount of custom code, or a more minimal library — unnecessary dependencies add transitive dependency weight, license obligations, and long-term maintenance cost that compound over time. (https://oneuptime.com/blog/post/2026-01-24-fix-dependency-management-issues/view)

### future-proofing

The future-proofing agent checks whether the plan's structural decisions — modularity, naming, folder layout, and documentation — will leave the codebase maintainable and handoff-ready for a developer who was not present when it was built.

1. Consistent naming conventions reduce the cognitive overhead of reading unfamiliar code and are the single highest-leverage structural investment for long-term maintainability; plans that introduce inconsistent names (different terms for the same concept across sessions) should be flagged before execution because names are expensive to change after the fact. (https://camphouse.io/blog/naming-conventions)
2. Single-responsibility modularity — components that do exactly one thing and are independently replaceable — is the foundation of a testable and maintainable codebase; plans that describe components with multiple responsibilities should be split before execution, not refactored after. (https://www.secoda.co/blog/how-to-ensure-modularity-and-reusability-in-data-pipelines)
3. Non-obvious architectural decisions made without documentation are the most common handoff failure mode; any session that introduces a new abstraction, changes an existing pattern, or deviates from established conventions must include a "Docs to update" entry that explains the why, not just the what. (https://typoapp.io/blog/crafting-an-effective-code-review-checklist)
4. File and folder placement that is inconsistent with where similar files live creates a codebase that is hard to navigate and forces new developers to search rather than predict where things are; reviewers must check each new file's proposed location against the existing structure. (https://www.nucamp.co/blog/coding-bootcamp-full-stack-web-and-mobile-development-what-are-the-best-practices-for-code-organization-in-large-projects)
5. Inline comments should explain the "why" behind non-obvious decisions, not restate what the code does; plans whose documentation sections are vague or missing a link to external context produce codebases where the next developer must reverse-engineer intent from implementation. (https://devcom.com/tech-blog/coding-standards-and-best-practices-guide-implementation-tips/)
