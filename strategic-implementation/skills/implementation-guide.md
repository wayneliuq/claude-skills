# Implementation Guide (Sessionized Plan)

This is the sessionized form of the implementation plan. It is produced by the `sessionize` skill after the user has approved the specification document. It is the document all agents review, and the document `session-plan` references when building individual session execution plans.

It is distinct from the specification document (produced by `implementation-drafter`) — the spec defines what and why; this document defines what gets built in which order, at what size, verified how.

---

## Authoring Rules (for the sessionize skill)

1. **One clear goal per session.** If you can't state the goal in one sentence, the session is doing too much — split it.
2. **Each session is self-contained.** A developer reading only that session block should know exactly what to build, which files to touch, and how to verify it — without reading other sessions.
3. **Every deliverable has a test.** No deliverable without a verification method. Tests should be specific enough to run.
4. **Size estimates are required.** S = <200 LOC, M = 200–500 LOC, L = 500–1000 LOC. Sessions estimated larger than L must be split before this document is presented to agents.
5. **Fail-fast ordering.** Sessions that validate the riskiest assumptions come first where possible. Do not build on unvalidated foundations.
6. **Hard decisions are locked.** Every `[HARD DECISION]` from the specification document must be reflected correctly in the sessions that implement it.
7. **Session Order & Dependencies is generated by scope-limiter.** Do not pre-fill this section.
8. **Success criteria coverage.** Every success criterion from the spec's Section 3 must be verifiable by at least one session's deliverables.
9. **Open Questions are true blockers only.** If listed, the plan cannot be executed until they are resolved.

## Template

```markdown
# Implementation Plan: [Feature / Change Name]
_Derived from: Implementation Spec — [spec name]_
_Status: Draft / Approved | Last updated: [date]_

---

## Overview
- **What:** [one paragraph — what this plan builds, in plain language]
- **Why:** [brief reference to the spec's Context and rationale]
- **Out of scope:** [minimum two explicit exclusions with one-sentence rationale each]
- **Hard decisions:** [list any [HARD DECISION] items from the spec that affect session design]

---

## Sessions

### Session N: [Name]
**Goal:** [one sentence — what is achieved by the end of this session]

**Deliverables & Tests:**
- [ ] Deliverable: [what is produced]
  Test: [how it is verified — specific enough to run or observe]
- [ ] Deliverable: [what is produced]
  Test: [how it is verified]

**Files affected:** [specific file paths to be created or modified]
**Docs to update:** [docs that must be updated after this session, or "none"]
**Estimated size:** S (<200 LOC) | M (200–500) | L (500–1000)

---

[...repeat for each session...]

---

## Session Order & Dependencies
_(Generated by scope-limiter — not pre-filled)_

- **Session order:** [recommended execution sequence]
- **Blocking dependencies:**
  - Session N requires Session M complete because: [reason]
- **Sessions that can run in parallel:** [list pairs or groups, or "none"]

---

## Open Questions
_(Unresolved items that would change the plan — must be resolved before execution begins)_
- [ ] ...

_(If none, write "None — all decisions resolved.")_
```

---

## Quality Checks (sessionize skill applies before running agent panel)

- [ ] Every session has a one-sentence goal
- [ ] Every deliverable has a specific, executable or observable test
- [ ] No session is estimated larger than L
- [ ] "Out of scope" has at least two entries
- [ ] All hard decisions from the spec are correctly reflected in sessions
- [ ] Every success criterion from the spec is covered by at least one session's deliverables
- [ ] Session Order & Dependencies is generated by scope-limiter (not empty, not pre-filled)
- [ ] Naming is consistent throughout (same concept, same name, everywhere)
- [ ] No session references a file or module not introduced earlier in the plan or already existing
- [ ] Open Questions contains only true blockers, or explicitly states "None"
