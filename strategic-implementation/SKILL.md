# strategic-implementation

A structured planning-to-execution workflow for software development. No code is written until a spec is approved. This skill is the orchestrator — it runs sub-skills and agents, synthesizes their outputs, and presents decisions to the user at each gate.

---

## Trigger

Invoke this skill when the user describes work they want to do — before any code, scaffolding, or plan is written.

---

## Entry: Two Paths

Before anything else, ask:

**"Do you have an existing implementation guide for this work?"**

- **Yes — implementation guide exists:** Ask the user to provide it (file path or paste contents). Skip directly to Step 8 (Build Session Plan). Steps 1–7 do not run.
- **No — no implementation guide:** Proceed to Step 1 (Clarify). Run the full workflow through Steps 1–7 to generate one.

---

## Step 1 — Clarify

Run `skills/clarify.md`.

List assumptions. Ask only what would change the approach. Wait for user confirmation before proceeding.

---

## Step 2 — Scope Assessment

After clarification, assess scope using ALL THREE criteria:

**Fast-path IF all three are true:**
1. Deliverable describable in one sentence
2. Change contained to one area (one component, file group, or feature)
3. No new architectural patterns or dependencies introduced

**If fast-path:** invoke `_copied-skills/writing-plans.md`, then `_copied-skills/executing-plans.md`. Do not continue this workflow.

**Otherwise:** continue to Step 3.

---

## Step 3 — Parallel Review: Architecture + UX/PMF

Launch both agents in parallel using the Agent tool:

**Agent A:** `skills/architecture-review.md`
**Agent B:** `skills/ux-pmf-review.md`

Wait for both to return before proceeding.

**If either returns STATUS: BLOCK:**
- Surface the blocking issue to the user with the agent's recommended next step.
- Do not proceed until resolved.

---

## Step 4 — Draft Implementation Guide

Using both agent outputs, draft the implementation guide per `skills/implementation-guide.md`.

At this point:
- Do NOT fill in "Session Order & Dependencies" — that is generated by scope-limiter in Step 5.
- Do NOT surface the draft to the user yet.

**If the change requires new architectural components or significant adaptation:**
→ BLOCK. Inform the user: "This change requires architectural decisions not yet captured in the architecture document. Please update the architecture doc first."
Do not proceed until resolved.

---

## Step 5 — Parallel Review: All Agents

### 5a — Collect Document References (before launching agents)

Before launching agents, ask the user for two document locations if not already known:

1. **Security policy document** — "Is there a security policy document for this project? If so, where is it?" (Pass location to `agents/security.md`; if none, pass that information so the agent can flag it.)
2. **Schema design / ERD** — "Is there a schema design document, entity-relationship diagram, or data dictionary? If so, where is it?" (Pass location to `agents/data-model.md`; if none, pass that information so the agent can assess whether to block or flag.)

Only ask if not already provided during clarification. Do not ask both at once if only one is relevant to the change (e.g., skip schema question for changes with no data storage).

### 5b — Launch Agents in Parallel

Launch all always-on agents in parallel using the Agent tool:

**Always launch:**
- `agents/10k-foot.md`
- `agents/technical-expert.md`
- `agents/scope-limiter.md`
- `agents/future-proofing.md`
- `agents/security.md` (pass: security policy doc location, or note that none exists)
- `agents/data-model.md` (pass: schema doc location, or note that none exists)
- `agents/api-contract.md`
- `agents/test-coverage.md`
- `agents/performance.md`
- `agents/dependency.md`

**Launch conditionally (if change involves UI, UX, or front-end):**
- `agents/frontend-engineer.md`

Each agent receives the full implementation guide draft as input.

Wait for all agents to return.

---

## Step 6 — Synthesize Agent Outputs

Read all agent outputs. Each uses the standard format:

```
## [Agent Name]
STATUS: PASS | FLAG | BLOCK
FLAGS:
  - ...
RECOMMENDATIONS:
  - [recommendation] — [rationale]
QUESTIONS FOR USER:
  - ...
```

**Rules:**
1. Collect all FLAGS and RECOMMENDATIONS.
2. Discard low-utility suggestions (vague, speculative, or not actionable in this plan).
3. Reconcile conflicts between agents — choose the more conservative or correct position; note the conflict.
4. If anything remains unclear after reconciliation → surface to user WITH a recommendation and rationale. Wait for response.
5. Apply the final patch list to the implementation guide.
6. Append the scope-limiter's session order + dependency map to the guide under "Session Order & Dependencies".
7. Run a final consistency check: does every session reference correct files, correct prior sessions, and consistent naming throughout?

---

## Step 7 — Present Implementation Guide

Present the full implementation guide to the user. Ask for approval or changes.

If the user requests changes: revise and re-present. Do not re-run all agents unless the changes are substantial enough to warrant re-review (use judgment).

Once approved: proceed to Step 8.

---

## Step 8 — Build Session Plan (Next Session Only)

Run `skills/session-plan.md` for the first session in the approved implementation guide.

The session plan covers **only the next session** — not the full implementation.

Constraint: the session plan must target ≤ ~1000 LOC edited or written.

---

## Step 9 — Parallel Review: Session Plan

Launch the same always-on agents again, now reviewing the session plan:
- `agents/10k-foot.md`
- `agents/technical-expert.md`
- `agents/scope-limiter.md`
- `agents/future-proofing.md`
- `agents/security.md`
- `agents/data-model.md`
- `agents/api-contract.md`
- `agents/test-coverage.md`
- `agents/performance.md`
- `agents/dependency.md`
- `agents/frontend-engineer.md` (if applicable)

Each agent receives both the session plan AND the full implementation guide as context. They must check alignment between the two.

Synthesize outputs using the same rules as Step 6.

---

## Step 10 — Present Session Plan

Present the final session plan to the user.

If the user requests changes: revise and re-present.

**Once approved: automatically invoke `_copied-skills/executing-plans.md`.** No manual step needed.

---

## Constraints (apply throughout)

- No code before spec. This workflow is a gate, not a shortcut.
- Agents return structured output only — no free prose. Orchestrator synthesizes.
- Agent outputs are bounded: max 5 flags, 1-sentence rationale each.
- Dependencies in the implementation guide are never pre-filled — scope-limiter generates them.
- Architecture document must exist and be current before any plan proceeds.
- UX/PMF check auto-skips for backend-only changes.
- Each session plan targets ≤ ~1000 LOC.
